## Decisões principais

### 1. Segurança: dropar a coluna `email` em vez de revogar privilégio
A policy `profiles_select_all using (true)` deixava qualquer usuário autenticado ler o e-mail de todos os outros via PostgREST. **Opções consideradas**:
- (A) Revogar `SELECT` na coluna `email` para `authenticated` — quebra `select("*")` em `AuthProvider.fetchProfile` (lê o próprio perfil) e em `AdminPage` (admin lista usuários);
- (B) Dropar a coluna `email` de `public.profiles`, já que o e-mail vive em `auth.users` (verdade única) e o cliente lê o próprio via `session.user.email`.

**Escolhida (B)**. Adicionalmente, criada RPC `admin_list_users()` (`SECURITY DEFINER`, restrita a `app_admin`) que lê o e-mail de `auth.users` para preservar a funcionalidade da aba *Usuários* do admin.

### 2. Modelo da fila: HTTP RPC + polling, **nunca Realtime**
A fila precisa funcionar exatamente quando o Realtime está saturado. Por isso o portão (`request_access`/`heartbeat_access`/`release_access`) usa só RPC via PostgREST, e o cliente faz polling no estado de espera. **Realtime é o recurso protegido — não pode ser dependência do mecanismo de proteção.**

### 3. Fail-open por padrão
Tanto a RPC (`enabled=false` ⇒ admite na hora) quanto o `AccessGate` no frontend (qualquer erro de RPC ⇒ admite). A fila **nunca pode virar um apagão**: prefiro permitir acesso indevido em caso de bug a derrubar usuários legítimos por falha no portão.

### 4. Concorrência: serialização da admissão via advisory lock
`pg_advisory_xact_lock(hashtext('resultadismo_access'))` no início de `request_access` garante que, sob rajada, o teto `max_active` não estoura. Liberação automática ao fim da transação.

### 5. Não evitar sessões já admitidas ao baixar o cap
`heartbeat_access` não re-checa o cap; só atualiza `last_seen_at`. Lógica explícita: **diminuir `max_active` segura novos acessos, sem expulsar quem está dentro.** Foi o que permitiu testar a UI da sala de espera sem perturbar as outras sessões ativas.

### 6. Fase 1 — índice covering em vez de materialização total da classificação
Materializar `league_standings` é a otimização "correta" no longo prazo, mas é cirurgia: depende de pontos por liga, `starts_on` por LC e jokers — fácil introduzir bug sutil de placar. **Decidido**: por ora, índice covering parcial em `predictions(match_id) where score_type is not null` + índice parcial em `matches(competition_id) where status='finished'`. Ganho real, risco zero, comportamento idêntico. Materialização fica documentada para quando **uma liga** passar de milhares de membros.

### 7. NÃO rodar `supabase db push` local — **a confusão mais perigosa da sessão**
O CLI desta máquina está logado numa conta que só enxerga `rezende-advocacia-dash` (`ynvscgiikwpgprebxbbl`). Se eu tivesse rodado `supabase link` + `db push` apontando para o Resultadismo, ele teria caído no único projeto visível — o errado — e o **drop da coluna `email` teria sido aplicado num dashboard de advocacia**. Decisão: **deploy em prod é exclusivamente via push para `main`** (integração Supabase↔GitHub aplica migrations; GitHub Action faz deploy das edge functions). Documentado em memória.

### 8. Auto-deploy de edge functions via GitHub Action
A integração Supabase↔GitHub aplica **migrations** automaticamente, mas **edge functions** historicamente precisavam de `supabase functions deploy` manual. O screenshot do dashboard mostrou `send-push` e `sync-football` deployadas em horários diferentes (17h e 16h atrás), indicando que o batch novo do `sync-football` provavelmente ainda não tinha subido. ## Cuidados aprendidos (referência rápida para futuras sessões)

- **O repositório é editado em paralelo** por outras sessões do Claude Code (frentes A/B/C, moderação, perfil público, etc. estavam em paralelo). Antes de commitar: re-rodar `git status`, conferir `git diff HEAD` em cada arquivo que se vai stagear, e **stage explícito por arquivo** (nunca `git add -A`).

- **NÃO rodar `supabase db push` local** sob nenhuma circunstância — esta máquina vê apenas `rezende-advocacia-dash`. Deploy de banco = push para `main` (integração GitHub aplica).
